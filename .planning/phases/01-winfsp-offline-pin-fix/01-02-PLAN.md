<objective>
Protect pinned items from memory cache eviction and populate SQLite metadata during pin_folder to ensure complete offline directory listings.

Purpose: The root cause of the Explorer hang is pinned items being evicted from memory cache, causing `find_child` to fall through to Graph API which blocks the VFS thread. This plan prevents that fallthrough by (1) protecting pinned items from eviction in memory cache (mirroring what `DiskCache` already does), and (2) ensuring `recursive_download` populates SQLite with the full directory tree so offline `find_child` and `list_children` always have metadata available.

Output: Modified `memory.rs` (eviction filter), modified `manager.rs` (wiring), modified `offline.rs` (SQLite population during pin).
</objective>

**Step 2: Modify `maybe_evict` to skip protected entries (memory.rs)**

In `maybe_evict()`, after collecting and sorting entries, filter out protected ones before eviction:

```rust
fn maybe_evict(&self) {
    if self.entries.len() <= MAX_ENTRIES {
        return;
    }

    let filter = self.eviction_filter.read().unwrap().clone();

    let mut entries: Vec<(u64, Instant)> = self
        .entries
        .iter()
        .map(|e| (*e.key(), e.value().last_access))
        .collect();

    entries.sort_by_key(|(_, t)| *t);

    let to_remove = entries.len() - EVICT_TO;
    let mut removed = 0;
    for (inode, _) in entries {
        if removed >= to_remove {
            break;
        }
        if let Some(ref f) = filter {
            if f(inode) {
                continue;  // protected — skip
            }
        }
        self.entries.remove(&inode);
        removed += 1;
    }
}
```

**Step 3: Modify `get` and `get_children` to bypass TTL for protected entries (memory.rs)**

In `get()` (line 31-41), after the TTL check fails, check if the entry is protected before removing:

```rust
pub fn get(&self, inode: u64) -> Option<DriveItem> {
    let mut entry = self.entries.get_mut(&inode)?;
    let elapsed = entry.inserted_at.elapsed().as_secs();
    if elapsed > self.ttl_secs {
        // Check if protected before evicting
        let protected = self.eviction_filter.read().unwrap()
            .as_ref()
            .is_some_and(|f| f(inode));
        if !protected {
            drop(entry);
            self.entries.remove(&inode);
            return None;
        }
        // Protected: refresh timestamps so it doesn't keep triggering TTL check
        entry.inserted_at = Instant::now();
    }
    entry.last_access = Instant::now();
    Some(entry.item.clone())
}
```

Modify `recursive_download` in `offline.rs` to persist each discovered item's metadata into SQLite during the download walk. This ensures that when the VFS later performs `find_child` or `list_children` while offline, the SQLite fallback tier has the full directory tree for the pinned folder.

**The inode problem:** `SqliteStore::upsert_item` requires `inode` and `parent_inode` parameters. `recursive_download` doesn't have access to `InodeTable` (it's in `carminedesktop-vfs`, which depends on `carminedesktop-cache` — circular dependency). 

**Solution:** Use a simple `AtomicU64` counter to assign temporary inodes during pin. These placeholder inodes are sufficient because:
1. `SqliteStore` uses `item_id` as the unique key (not inode) — `ON CONFLICT(item_id) DO UPDATE`
2. When the VFS later discovers these items via `find_child`, it calls `InodeTable::allocate(&item.id)` which assigns the real inode
3. The `get_children(parent_inode)` query uses `parent_inode`, so parent-child relationships must be internally consistent within the recursive walk, but don't need to match VFS inodes
4. If the item already exists in SQLite (from a previous browse), the upsert updates metadata but preserves the existing inode
