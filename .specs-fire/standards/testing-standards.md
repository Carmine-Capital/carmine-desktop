# Testing Standards

## Overview

Integration-style tests in `crates/<name>/tests/`. No inline `#[cfg(test)]` modules. Async tests via `#[tokio::test]`. HTTP mocking via `wiremock`. All tests return `cloudmount_core::Result<()>` for clean `?` propagation.

## Testing Framework

**Framework**: Rust built-in (`#[test]`, `#[tokio::test]`)
**Runner**: `cargo test --all-targets`

## Test Types

| Type | Tool | Location | When to Use |
|------|------|----------|-------------|
| Integration | `#[tokio::test]` | `crates/<name>/tests/` | All tests — the primary pattern |
| HTTP mock | `wiremock` | `crates/<name>/tests/` | Graph API client tests |
| Time-sensitive | `tokio::time::pause()` | inline in test | Retry/backoff determinism |

## Coverage Requirements

**Target**: Best-effort (no enforced threshold in CI currently)
**Enforcement**: Manual review

**Critical paths that MUST have coverage:**
- OAuth2 token refresh flow
- Graph API retry/backoff behavior
- Cache eviction and writeback
- Config merge (user config + built-in defaults)
- VFS read/write operations

## Test Naming

**Pattern**: `test_<module>_<operation>_<scenario>()` (cache/auth) or `<operation>_<scenario>()` (graph)

**Examples**:
- `test_config_merge_user_overrides_defaults` — config merge with user values
- `test_cache_eviction_triggers_at_max_entries` — LRU eviction at limit
- `get_drive_item_retries_on_throttle` — Graph client retry on 429
- `upload_file_uses_session_for_large_files` — upload session threshold

## Test Structure

```rust
#[tokio::test]
async fn test_module_operation_scenario() -> cloudmount_core::Result<()> {
    // Arrange
    let mock_server = MockServer::start().await;
    Mock::given(method("GET"))
        .and(path("/v1.0/me/drive/items/root"))
        .respond_with(ResponseTemplate::new(200).set_body_json(&item))
        .mount(&mock_server)
        .await;

    let client = GraphClient::new(mock_server.uri(), token());

    // Act
    let result = client.get_root_item().await?;

    // Assert
    assert_eq!(result.id, "root");
    Ok(())
}
```

## Mock Strategy

**Approach**: `wiremock` for HTTP mocking — real HTTP stack, no trait injection needed

**Guidelines**:
- Use `MockServer::start().await` — binds to random port, no conflicts
- Mount mocks before the code under test runs
- Use `Mock::given(method(...)).and(path(...))` for specific matching
- For retry tests: use `tokio::time::pause()` + `tokio::time::advance()` for deterministic timing
- File I/O tests: use `std::env::temp_dir()` with explicit cleanup before each test run

## Test Data

**Strategy**: Inline JSON literals / helper constructors in test files

**Guidelines**:
- Build test fixtures close to where they're used
- Use `serde_json::json!()` macro for Graph API response payloads
- Clean up temp files at the START of each test (not just at end — handles previous test crashes)
- No shared global test state — each test is independent

## Running Tests

> **Linux (immutable distros)**: All commands must be run inside the `cloudmount-build` toolbox container. See `tech-stack.md` → Linux Development Environment for setup instructions.

### Linux (via toolbox)

```bash
# Run all tests
toolbox run --container cloudmount-build cargo test --all-targets

# Run specific crate
toolbox run --container cloudmount-build cargo test -p cloudmount-cache

# Run specific test
toolbox run --container cloudmount-build cargo test -p cloudmount-graph test_get_drive_item

# Check formatting + lint first
toolbox run --container cloudmount-build cargo fmt --all -- --check
toolbox run --container cloudmount-build cargo clippy --all-targets --all-features
```

Or enter the container once and run commands directly:

```bash
toolbox enter cloudmount-build
# now inside the container:
cargo test --all-targets
```

### macOS / Windows (native)

```bash
# Run all tests
cargo test --all-targets

# Run specific crate
cargo test -p cloudmount-cache

# Run specific test
cargo test -p cloudmount-graph test_get_drive_item

# Check formatting + lint first
cargo fmt --all -- --check && cargo clippy --all-targets --all-features
```

## CI/CD Integration

**Pipeline**: GitHub Actions (`.github/workflows/ci.yml`)
**Trigger**: Every push and pull request

**Required gates**:
- `cargo fmt --all -- --check` passes
- `cargo clippy --all-targets --all-features` passes (zero warnings)
- `cargo test --all-targets` passes on Linux, macOS, and Windows
